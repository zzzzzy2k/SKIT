# SKit PRD & WBS 改进设计文档

**版本**：v1.0
**日期**：2026-05-02
**基于**：需求文档.md v1.0 + WPS.md

---

## 1. 改进背景

原始 PRD 和 WBS 存在以下问题：
- 食材筛选逻辑存在歧义（AND vs OR）
- 人数缩放规则过于简化（线性缩放不适合调味料）
- 缺少错误状态和边界情况的定义
- 数据解析方案不够具体（HowToCook Markdown 格式不统一）
- WBS 阶段1 任务过于笼统，缺少依赖关系
- 缺少关键任务（错误处理、Loading 状态、空状态）

## 2. PRD 功能需求改进

### 2.1 F2 食材筛选 — 混合逻辑

**原描述**：只显示包含所有已选食材的菜谱。

**改进为**：

- **默认 AND 逻辑**：只显示同时包含所有已选食材的菜谱。
- **空结果降级**：如果 AND 筛选结果为空，自动切换为 OR 逻辑，并显示提示：「未找到同时包含这些食材的菜谱，以下菜谱至少包含其中一种食材」。
- **UI 反馈**：降级时结果列表顶部显示黄色提示条，用户可点击「仅显示完全匹配」切回 AND 逻辑。

### 2.2 F4 人数调整 — 分类缩放

**原描述**：等比例缩放所有食材的数量。

**改进为**：

- **主料**（肉、蔬菜、蛋、米面等）：严格线性缩放（人数 × 原始用量）。
- **调味料**（盐、糖、酱油、醋、料酒等）：按 `人数^0.7` 系数缩放，四舍五入到合理精度。
- **标记策略**：无法解析为数值的食材（如「适量」「少许」）保持原样，添加 `~` 前缀标记为「按口味调整」。
- **食材分类词典**：在 `fetch-recipes.js` 中维护一个调味料关键词列表，用于区分主料和调味料。

**调味料词典示例**：
```javascript
const SEASONINGS = [
  "盐", "糖", "酱油", "生抽", "老抽", "醋", "料酒", "蚝油",
  "花椒", "八角", "桂皮", "香叶", "辣椒", "胡椒", "五香粉", "十三香",
  "鸡精", "味精", "芝麻油", "香油", "淀粉", "生粉"
];
```

**缩放示例**（原始 1 人份 → 4 人份）：
| 食材 | 原始用量 | 缩放后 | 说明 |
|:--|:--|:--|:--|
| 鸡蛋 | 3个 | 12个 | 主料，线性缩放 (3×4) |
| 番茄 | 2个 | 8个 | 主料，线性缩放 (2×4) |
| 盐 | 3克 | 5克 | 调味料，系数缩放 (3×4^0.7≈5.3) |
| 适量 | — | ~适量 | 无法解析，标记为按口味调整 |

### 2.3 F6 购物清单 — 明确范围

- **v1.0**：仅支持基于当前查看的单个菜谱生成清单。
- **v2.0**：支持从收藏夹多选菜谱，累加合并食材生成一周采购清单。

### 2.4 离线能力 — 明确方案

- **v1.0**：数据已打包为本地 JSON，关闭网络后仍可正常使用（SPA 内导航不依赖网络）。刷新页面需联网加载 HTML/CSS/JS。
- **v2.0**：添加 Service Worker + PWA manifest，实现完整离线缓存。

## 3. 数据解析与数据结构

### 3.1 解析方案 — AST 解析

**工具**：使用 `marked` 或 `unified` + `remark` 将 Markdown 解析为 AST。

**解析策略**：
1. 遍历 AST，按 `heading` 节点分段（h2 作为主要分区标识）。
2. 识别关键分区：「必备原料和工具」「原材料」「操作步骤」「附加内容」等（支持多种标题变体）。
3. 从食材分区的列表节点（`list` / `listItem`）提取食材文本。
4. 从步骤分区的有序列表节点提取步骤文本。
5. 用正则从食材文本中解析「名称 + 数量 + 单位」（如「鸡蛋 3个」→ `{name: "鸡蛋", qty: 3, unit: "个"}`）。

**难度解析**：统计正文中 `★` 或 `⭐` 的数量，映射为 0-5 星。

**容错**：如果某个菜谱解析失败（缺少关键字段），记录到 `errors.log`，跳过该菜谱，不阻塞整体流程。

### 3.2 数据结构增强

在原始 PRD 基础上增加字段：

| 新增字段 | 类型 | 说明 |
|:--|:--|:--|
| `category` | string | 菜谱分类（从目录结构提取，如「荤菜」「素菜」「汤类」） |
| `servings` | number | 原始基准人数（默认 1，如果菜谱明确写了「2人份」则为 2） |
| `parsed_ingredients` | array | 结构化食材 `[{name, qty, unit, is_seasoning}]` |

**完整数据结构**：
```json
{
  "id": "西红柿炒鸡蛋",
  "title": "西红柿炒鸡蛋",
  "category": "荤菜",
  "servings": 1,
  "ingredients": ["鸡蛋", "番茄", "盐"],
  "ingredients_with_quantity": ["鸡蛋 3个", "番茄 2个", "盐 3克"],
  "parsed_ingredients": [
    {"name": "鸡蛋", "qty": 3, "unit": "个", "is_seasoning": false},
    {"name": "番茄", "qty": 2, "unit": "个", "is_seasoning": false},
    {"name": "盐", "qty": 3, "unit": "克", "is_seasoning": true}
  ],
  "steps": ["番茄切块...", "鸡蛋打散..."],
  "tools": ["炒锅", "碗"],
  "difficulty": 1,
  "raw_markdown": "..."
}
```

## 4. 错误处理与边界情况

### 4.1 错误状态定义

| 场景 | 处理方式 |
|:--|:--|
| `recipes.json` 加载失败 | 显示错误页面 + 「重试」按钮 |
| 食材筛选结果为空（AND 逻辑） | 自动降级为 OR 逻辑 + 黄色提示条 |
| 食材筛选结果为空（OR 逻辑也为空） | 显示「没有找到匹配的菜谱，试试减少食材筛选条件」 |
| 菜谱详情页 id 不存在 | 显示「菜谱未找到」+ 返回首页链接 |
| localStorage 不可用（隐私模式） | 收藏功能降级为内存存储，提示「收藏仅在本次会话有效」 |

### 4.2 Loading 状态

- **首页加载**：显示骨架屏（3-5 个卡片占位），模拟真实布局。
- **菜谱详情加载**：由于数据已在本地 JSON，理论上无需 loading。但如果数据量大（>500 道菜），首次解析可能有延迟，此时显示 spinner。

### 4.3 边界情况

- 食材列表为空（JSON 解析失败或无数据）：显示「暂无菜谱数据，请运行数据抓取脚本」。
- 人数输入非法值（0、负数、>10、非数字）：自动修正为最近的合法值（1 或 10）。
- 收藏列表为空：显示空状态插图 + 「去首页探索吧」引导。

在原始 PRD 基础上补充：

| 功能 | 验收条件 |
